# SD-BUBBLELENS-001-Epic4: Lightweight Demographic Survey

**Parent PRD**: PRD-BUBBLELENS-P1-001 (Epic 4)
**Epic**: Lightweight Demographic Survey
**Worker Type**: `frontend-dev-expert` (UI) + `backend-solutions-engineer` (API)

## Overview

A 3-field demographic survey embedded directly in the Chrome extension popup. Triggered once after the user's first successful feed capture. No login, no accounts -- responses attach to the anonymous browser UUID already generated by the extension. This is the critical bridge between anonymous feed data and demographically-tagged data that powers persona simulation.

## Architecture

```
Extension Popup (after first capture)
    │
    ├── Survey UI (3 fields inline)
    │   ├── Political Leaning (5-point scale)
    │   ├── Age Range (5 brackets)
    │   └── Gender/Orientation (6 options)
    │
    ├── Submit → POST /api/profile { browserId, politicalLeaning, ageRange, genderOrientation }
    │               │
    │               ▼
    │         Upsert demographic_profiles (linked to browsers table)
    │
    └── Skip → chrome.storage.local { surveyCompleted: true }

State Machine:
    NEVER_SHOWN → (first capture) → SHOWING → (submit/skip) → COMPLETED
    COMPLETED → (click "Update my profile") → EDITING → (submit) → COMPLETED
```

## Component Design

### Survey State Management

```typescript
// extension/src/lib/survey-state.ts
type SurveyState = 'never_shown' | 'showing' | 'completed' | 'skipped';

interface SurveyStorage {
  surveyState: SurveyState;
  captureCount: number;
  profileData?: ProfileData;
}

async function getSurveyState(): Promise<SurveyStorage> {
  const data = await chrome.storage.local.get('survey');
  return data.survey || { surveyState: 'never_shown', captureCount: 0 };
}

async function recordCapture(): Promise<SurveyState> {
  const state = await getSurveyState();
  state.captureCount++;

  // Trigger survey after first capture
  if (state.captureCount === 1 && state.surveyState === 'never_shown') {
    state.surveyState = 'showing';
  }

  await chrome.storage.local.set({ survey: state });
  return state.surveyState;
}

async function completeSurvey(data: ProfileData | null): Promise<void> {
  const state = await getSurveyState();
  state.surveyState = data ? 'completed' : 'skipped';
  state.profileData = data || undefined;
  await chrome.storage.local.set({ survey: state });
}
```

async function submitProfile(
  browserId: string,
  profile: ProfileData
): Promise<boolean> {
  try {
    const response = await fetch(`${API_BASE_URL}/api/profile`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },

### Profile API Route

```typescript
// src/app/api/profile/route.ts
import { z } from 'zod';
import { db } from '@/lib/db';

const profileSchema = z.object({
  browserId: z.string().uuid(),
  politicalLeaning: z.number().int().min(1).max(5).optional(),
  ageRange: z.enum(['18-24', '25-34', '35-44', '45-54', '55+']).optional(),
  genderOrientation: z.enum([
    'straight_male', 'straight_female', 'gay_lesbian',
    'bisexual', 'nonbinary_other',
  ]).optional(),
});

export async function POST(req: Request) {
  const body = await req.json();
  const result = profileSchema.safeParse(body);
  if (!result.success) {
    return Response.json({ error: result.error.flatten() }, { status: 400 });
  }

  const { browserId, ...profileData } = result.data;

  // Find browser record (must exist from prior feed capture)
  const browser = await db.browser.findUnique({ where: { browserId } });
  if (!browser) {
    return Response.json({ error: 'Browser not found. Capture a feed first.' }, { status: 404 });
  }

  // Upsert demographic profile
  const profile = await db.demographicProfile.upsert({
    where: { browserId: browser.id },
    create: {
      browserId: browser.id,
      politicalLeaning: profileData.politicalLeaning ?? null,
      ageRange: profileData.ageRange ?? null,
      genderOrientation: profileData.genderOrientation ?? null,
      isSeed: false,
    },
    update: {
      politicalLeaning: profileData.politicalLeaning ?? null,
      ageRange: profileData.ageRange ?? null,
      genderOrientation: profileData.genderOrientation ?? null,
      updatedAt: new Date(),
    },
  });

  return Response.json({ id: profile.id }, { status: 200 });
}

// GET /api/profile?browserId=xxx - retrieve profile for pre-fill
export async function GET(req: Request) {
  const url = new URL(req.url);
  const browserId = url.searchParams.get('browserId');
  if (!browserId) return Response.json({ error: 'browserId required' }, { status: 400 });

  const browser = await db.browser.findUnique({ where: { browserId } });
  if (!browser) return Response.json(null, { status: 404 });

  const profile = await db.demographicProfile.findUnique({
    where: { browserId: browser.id },
    select: {
      politicalLeaning: true,
      ageRange: true,
      genderOrientation: true,
    },
  });

  return Response.json(profile);
}
```

## Data Model

Updates the `DemographicProfile` model in the Prisma schema (simplified from the original 7-field version):

```prisma
model DemographicProfile {
  id                String   @id @default(uuid()) @db.Uuid
  browserId         String   @unique @map("browser_id") @db.Uuid
  politicalLeaning  Int?     @map("political_leaning") @db.SmallInt // 1-5
  ageRange          String?  @map("age_range")                      // '18-24' etc.
  genderOrientation String?  @map("gender_orientation")             // 'straight_male' etc.
  isSeed            Boolean  @default(false) @map("is_seed")
  updatedAt         DateTime @default(now()) @updatedAt @map("updated_at") @db.Timestamptz

  browser Browser @relation(fields: [browserId], references: [id], onDelete: Cascade)

  @@map("demographic_profiles")
}
```

**Design decision**: Combined gender + sexual orientation into a single `genderOrientation` field. This reduces the survey from 4 fields to 3 while capturing the signals most impactful for filter bubble analysis (YouTube's algorithm treats content differently based on these combined signals). The full 7-field survey with separate gender, orientation, ethnicity, country, and interests is deferred to post-MVP when Clerk auth enables a proper onboarding flow.

## API Design

| Method | Path | Description | Auth | Rate Limit |
|--------|------|-------------|------|-----------|
| POST | `/api/profile` | Create or update demographic profile | None (browser UUID) | 5/hour/IP |
| GET | `/api/profile?browserId=X` | Retrieve profile for pre-fill | None | None |

**Error responses**:

| Status | Body | When |
|--------|------|------|
| 400 | `{ error: { fieldErrors: {...} } }` | Validation failure |
| 404 | `{ error: "Browser not found" }` | No browser record for given UUID (capture a feed first) |

## Testing Strategy

- **Unit tests**: Survey state machine (never_shown -> showing -> completed/skipped)
- **Unit tests**: Zod schema validation (valid/invalid payloads, all-optional submission)
- **Unit tests**: Survey only triggers on first capture (not second, third, etc.)
- **Integration tests**: POST profile with valid data creates record
- **Integration tests**: POST profile with same browserId updates existing record (upsert)
- **Integration tests**: POST profile with unknown browserId returns 404
- **Integration tests**: GET profile returns saved data for pre-fill
- **Component tests**: Survey renders all 3 fields with correct options
- **Component tests**: Skip button hides survey and records skip state
- **Component tests**: "Update my profile" shows pre-filled form
